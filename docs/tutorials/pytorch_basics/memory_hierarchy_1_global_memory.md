# GPU 内存层级 1：Global Memory (显存) —— 那个遥远而巨大的仓库

这是“GPU 内存层级四部曲”的第一篇。我们要讲的是 **Global Memory**（也就是显存 VRAM）。

## 1. 它是谁？
当你买显卡时看到的“24GB 显存”，指的就是 Global Memory。它是 GPU 上容量最大的存储空间，所有 SM（Streaming Multiprocessor）都可以访问它。

*   **比喻**：这是离工厂（SM）几十公里远的**市中心大仓库**。
*   **特点**：容量大，但读写非常慢（延迟高），路非常窄（带宽有限）。

## 2. 核心问题：Memory Coalescing (合并访问)

想象一下，仓库门口只有一条很窄的路。如果 32 辆卡车（Warp 中的 32 个线程）要进货：

### 2.1 理想情况：合并访问 (Coalesced)
*   **场景**：32 个线程正好要读显存中**连续**的 128 字节（比如 float32 数组的第 0 到 31 个元素）。
*   **动作**：显存控制器极其聪明，它发现这 32 个请求是连续的，于是**只发出一辆大卡车**，一次性就把这 128 字节拉回来了。
*   **结果**：带宽利用率 100%，速度极快。

### 2.2 糟糕情况：非合并访问 (Uncoalesced)
*   **场景**：32 个线程要读显存中**分散**的数据（比如第 0, 100, 200... 个元素）。这叫“跳跃访问” (Strided Access)。
*   **动作**：显存控制器没办法合并，只能**派出 32 辆小车**，每辆车只拉 4 个字节回来。
*   **结果**：虽然总数据量一样，但发车的次数多了 32 倍！路瞬间堵死，带宽利用率可能只有 3%。

---

## 3. 代码示例：Triton 中的连续 vs 跳跃

为了让您直观感受，我们用 Triton 伪代码演示这两种访问模式。

### 3.1 连续加载 (快)
这段代码演示了如何让 1024 个线程同时去读内存里**紧挨着的** 1024 个数。

```python
# 1. 获取当前程序块(Block)的 ID
# 想象我们把任务分成了很多块，现在我是第 pid 块
pid = tl.program_id(0)

# 2. 计算当前块负责的数据起始位置
# 假设每个 Block 负责处理 1024 个元素
# 第 0 块负责 0-1023，第 1 块负责 1024-2047，以此类推
block_start = pid * 1024

# 3. 生成连续的偏移量 (关键步骤！)
# tl.arange(0, 1024) 会生成一个向量：[0, 1, 2, ..., 1023]
# 加上 block_start 后，比如 pid=0，offsets 就是 [0, 1, 2, ..., 1023]
offsets = block_start + tl.arange(0, 1024)

# 4. 执行加载 (Load)
# ptr 是显存中数据的首地址 (指针)
# ptr + offsets 意味着我们要访问的地址是：ptr+0, ptr+1, ptr+2 ... ptr+1023
# ！！注意！！：因为这些地址是连续的，GPU 显存控制器会把它们“合并”成几个大请求
# 就像开一辆大卡车一次拉走，而不是开 1024 辆小摩托车
data = tl.load(ptr + offsets)
```

#### 通俗解读：食堂打饭
*   **连续加载 (Coalesced)**：
    *   **场景**：32 个学生（线程）排队打饭。
    *   **动作**：大家**挨个儿**站好，阿姨（显存控制器）只需要拿起勺子，**顺手一滑**，就把 32 个人的饭盒都装满了。
    *   **效率**：极高，一气呵成。

*   **跳跃加载 (Uncoalesced)**：
    *   **场景**：32 个学生想打饭，但是他们**分散**在食堂的各个角落（一个在门口，一个在窗边，一个在二楼...）。
    *   **动作**：阿姨必须**跑**到门口给第一个人打饭，再**跑**到窗边给第二个人打饭...
    *   **效率**：极低，阿姨大部分时间都在跑路（寻址），而不是打饭（传输数据）。

### 3.2 跳跃加载 (慢)
```python
# 假设我们要转置一个矩阵，或者读一个稀疏矩阵
# 生成跳跃的偏移量：[0, 100, 200, ..., 102300]
offsets = block_start + tl.arange(0, 1024) * 100

# Load：这会触发非合并访问，非常慢！
# 显存控制器必须发起 1024 次独立的读取请求
data = tl.load(ptr + offsets)
```

## 4. 优化建议

1.  **尽量让数据在内存中连续存放**：比如在 PyTorch 中使用 `contiguous()`。
2.  **避免随机访问**：如果算法必须随机访问（比如 Graph Neural Networks），尝试先对图进行重排序（Reordering），让邻居节点在内存中尽量靠得近一些。
3.  **使用 Shared Memory 做缓冲**：先把乱序的数据搬到 Shared Memory 里整理好，再统一写回 Global Memory。
